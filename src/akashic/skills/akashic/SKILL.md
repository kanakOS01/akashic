# Akashic Knowledge Base

Akashic keeps Git-backed knowledge repositories generated and maintained as
plain Markdown. Before answering questions about system architecture, service
ownership, business entities, cross-service flows, or past architecture
decisions, select a knowledge base reference path first.

## Selection rule

Do not assume the current working directory is the knowledge base. Do not inspect
the current source repository as the Akashic knowledge base. First show the user
the available Akashic knowledge base reference paths from the global Akashic
folder/registry, then ask which one to use when there is more than one.

## Where the knowledge repo lives

The knowledge repo is a normal Git repository, separate from the source repositories
it documents. After the user chooses a reference path, use that path as the
knowledge repo root. The attached source repos it covers are listed under
`repositories:` in `.akashic/config.yaml`.
Per-machine local paths for those source repos live in the gitignored
`.akashic/config.local.yaml`.

## Folder structure and what each directory means

```
knowledge/
├── .akashic/           Akashic config, cache, and logs (not documentation)
├── services/           One directory per service
├── flows/              End-to-end business workflows
├── system/             Cross-cutting / system-wide documentation
├── adr/                Architecture Decision Records
├── entities/           Business entities
├── glossary/           Shared terminology
└── README.md
```

- **`services/<name>/`** — one directory per attached service, typically containing:
  - `purpose.md` — what the service does and why it exists
  - `architecture.md` — internal structure, key components, dependencies
  - `contracts.md` — APIs, events, schemas the service exposes or consumes
  - `workflows.md` — notable internal workflows specific to this service
  Start here when asked "what does service X do" or "how is X structured."

- **`flows/`** — one file per end-to-end business workflow that spans multiple
  services (e.g. `checkout.md`, `customer-login.md`). Read these when asked how a
  process works across the system, not just within one service.

- **`system/`** — cross-cutting docs that don't belong to a single service, e.g.
  `service-map.md` (what services exist and how they relate) and
  `interactions.md` (how services call each other). Good starting point for any
  "how does the system fit together" question.

- **`adr/`** — Architecture Decision Records, one per significant decision
  (e.g. `ADR-001-dual-write.md`). Read these when asked "why was X built this way"
  or before proposing a change that might contradict a prior decision.

- **`entities/`** — one file per core business entity/domain object (e.g.
  `Order.md`, `Coupon.md`). Read these when asked what a business term or object
  means, or which services touch it.

- **`glossary/`** — shared terminology definitions. Check here for unfamiliar
  domain or project-specific terms before assuming a meaning.

## How to navigate it

1. Identify what kind of question is being asked: single-service internals →
   `services/<name>/`; multi-service process → `flows/`; system-wide shape →
   `system/`; historical rationale → `adr/`; a business concept/term →
   `entities/` or `glossary/`.
2. Read the relevant file(s) fully before answering — these documents are meant
   to be the authoritative starting point, more reliable than re-deriving the
   answer from scratch by reading source code.
3. Cross-reference: a flow may reference several services; a service's
   `contracts.md` may reference entities. Follow those references as needed.

## Document format and human edits

Every generated document starts with frontmatter (`generated`, `type`, `name`,
`sources`, `updated_at`, `generator`) identifying what was generated and from
where. Content between `<!-- HUMAN START -->` and `<!-- HUMAN END -->` markers
is manually authored by engineers and takes precedence over generated content —
treat it as more trustworthy than the generated sections around it.

## Keeping it current

This knowledge repo is regenerated by running `akashic generate` from within it.
If the documentation looks stale or contradicts the code, say so rather than
silently trusting one source over the other.
