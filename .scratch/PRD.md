# PRD: Akashic CLI (MVP)

> Status: Draft · Local document (not tracked in an issue tracker) · Target: v1 MVP

## Problem Statement

Engineering teams working across many distributed repositories have no shared, trustworthy, up-to-date knowledge layer that both humans and AI coding agents can rely on. Knowledge lives in people's heads, scattered wikis, and stale docs. When an AI agent (or a new engineer) needs to understand how services fit together, what a business entity means, or how a flow works end-to-end, there is nothing authoritative to read. Existing tools either lock knowledge into a proprietary backend/index or require moving repositories around.

Engineers want a knowledge base that:
- lives in plain Git (no proprietary store, no hidden database),
- works entirely on their machine (no cloud, no backend),
- is generated by the coding agents they already use (Claude Code, Codex) rather than a new bespoke code-understanding engine,
- lets them keep source repositories exactly where they are,
- and can be browsed as a documentation site.

## Solution

Akashic is a local-first, Git-backed CLI that turns a directory into a **knowledge repository** of structured Markdown, and orchestrates the user's existing coding agent to author that documentation.

The user runs `akashic init` to scaffold a Git repo, `akashic attach` to register their existing source repositories (without moving them), and `akashic generate` to open an **interactive** Claude Code / Codex session seeded with a composed prompt. The agent reads the attached repositories and writes structured docs (services, flows, system maps, entities, ADRs, glossary) directly into the knowledge repo. Everything is committed to Git by the user, browsable via `akashic serve` (MkDocs), and exportable as a static site via `akashic build-site`.

Akashic is a **launcher, prompt-composer, and validator** — it never asks an LLM directly and never authors documents itself in v1. The coding agent is the documentation author.

## User Stories

1. As an engineer, I want to initialize a knowledge repository with `akashic init`, so that I have a ready-to-use Git repo scaffolded with the standard knowledge layout.
2. As an engineer, I want `akashic init` to run `git init`, add a `.gitignore`, and make an initial commit, so that my knowledge base is version-controlled from the very first moment.
3. As an engineer, I want `init` to be idempotent, so that re-running it in an existing knowledge repo does not clobber my work or error out.
4. As an engineer, I want `akashic attach` (with no argument) to register the current directory, so that I can quickly add the repo I'm standing in.
5. As an engineer, I want `akashic attach /path/to/repo` to register a repo by path, so that I can add repositories from anywhere on disk.
6. As an engineer, I want `attach` to validate that the target is a Git repository, so that I don't accidentally register a non-repo directory.
7. As an engineer, I want the repository name to default to the directory's basename, so that I don't have to name every repo manually.
8. As an engineer, I want to override the name with `--name`, so that I can resolve collisions or use a clearer name.
9. As an engineer, I want `attach` to dedupe by resolved absolute path, so that attaching the same repo twice updates it in place instead of creating duplicates.
10. As an engineer, I want `attach` to error when a name collides with a different repo, so that generated folders never clash.
11. As an engineer, I want attaching to never modify my source repository, so that documentation work is completely non-invasive.
12. As an engineer, I want `akashic detach <name>` to remove a repo from configuration without deleting any files, so that I can safely stop tracking a repo.
13. As an engineer, I want `detach` to tell me the generated docs remain and how to remove them (`git rm`), so that I'm in control of cleanup.
14. As an engineer, I want `akashic list` to show all attached repositories, so that I can see what's registered at a glance.
15. As an engineer, I want `list` to indicate when an attached repo has no local path on this machine, so that I know it needs configuring before I can generate.
16. As an engineer, I want `akashic generate` to open an interactive agent session seeded with a full generation prompt, so that I can watch the agent work and approve permissions live.
17. As an engineer, I want the generation prompt to include the list of attached repositories and their paths, so that the agent knows exactly what to document.
18. As an engineer, I want the agent to fan out / use subagents on its own, so that I get parallel, thorough generation without Akashic having to orchestrate it in v1.
19. As an engineer, I want the agent to read my existing knowledge files and preserve anything between `<!-- HUMAN START -->` and `<!-- HUMAN END -->`, so that my manual documentation survives regeneration.
20. As an engineer, I want generated documents to carry frontmatter (`generated`, `type`, `name`, `sources`, `updated_at`, `generator`), so that both humans and tools can tell what was generated, from where, and when.
21. As an engineer, I want `generate` to expose each source repo to the agent read-side while writing only into the knowledge repo, so that my source code is documented but not mutated.
22. As an engineer, I want `generate` to perform no Git commits and no staging, so that I can review the diff and commit on my own terms.
23. As an engineer, I want `generate` to print a summary of changed files after the session ends, so that I immediately see what was produced.
24. As an engineer, I want `generate` to warn and skip any attached repo whose local path is missing, and only abort if no valid repos remain, so that one bad path doesn't block the whole run.
25. As an engineer, I want to choose my agent provider (Claude Code or Codex) in config, so that I can use whichever agent I prefer.
26. As an engineer, I want both Claude Code and Codex to work in v1, so that I'm not locked into one vendor.
27. As an engineer, I want Akashic to detect whether my chosen agent is installed and report its version, so that I get a clear error instead of a cryptic failure.
28. As an engineer, I want to override the agent binary path in config, so that non-standard installs still work.
29. As an engineer, I want to override any prompt template by dropping a file in `.akashic/custom-prompts/`, so that I can tailor generation to my team's conventions.
30. As an engineer, I want `akashic serve` to launch a local documentation site with navigation, search, Markdown rendering, and Mermaid diagrams, so that I can browse the knowledge base in a browser.
31. As an engineer, I want to edit documentation in my own editor and see it reflected on refresh, so that editing is friction-free (local + Git).
32. As an engineer, I want `akashic build-site` to compile the docs into a static `dist/`, so that I can deploy to GitHub Pages / Netlify / Cloudflare Pages.
33. As an engineer, I want `akashic doctor` to validate my config, repo paths, Git, agent installation, and that the knowledge repo is writable, so that I can diagnose setup problems quickly.
34. As an engineer, I want `akashic status` to show attached repos, last generation time, pending Git changes, and doc statistics, so that I understand the current state of my knowledge base.
35. As a teammate, I want to clone the knowledge repo and set my own local repo paths without touching the committed config, so that machine-specific paths never break for others.
36. As an engineer, I want to run Akashic commands from anywhere inside the knowledge repo (git-style discovery), so that I don't have to be in the exact root.
37. As an engineer, I want to override the knowledge repo location with `--knowledge` or `AKASHIC_HOME`, so that I can script against a specific repo.
38. As an AI coding agent, I want to consume the generated Markdown knowledge base, so that I can understand the system across repositories when assisting with development.
39. As an engineer, I want cache and logs to be gitignored, so that only meaningful knowledge is version-controlled.

## Implementation Decisions

### Language, packaging, distribution
- **Python 3.10+**, packaged via `pyproject.toml`, distributed via **pipx/uv**. Console entry point: `akashic = akashic.cli:app`.
- **Typer** for the CLI (type-hint driven; good help/UX).
- **Pydantic** models for config; config is validated on load with clear errors.
- **System `git` via subprocess** for all Git operations (git is already a hard dependency) — no GitPython.

### Package structure (deliberate seams for a future frontend swap)
```
akashic/
  cli/        # Typer command entrypoints — thin; delegate to engine
  engine/     # config, workspace + repo managers, prompt builder, AgentProvider(s), git ops, generation orchestration
  server/     # site layer: v1 MkDocs adapter behind a SiteProvider interface
  frontend/   # placeholder for a future React app (empty in v1)
  prompts/    # bundled modular prompt templates
```
- `cli/` and `engine/` must not depend on `server/` internals; the site is reached only through the `SiteProvider` interface so a FastAPI+React stack can replace MkDocs later without touching CLI or engine.

### Workspace & configuration
- **Discovery:** git-style walk up from cwd to find `.akashic/config.yaml`. Override via `--knowledge <path>` or `AKASHIC_HOME`.
- **Split config:**
  - `.akashic/config.yaml` — **committed.** Portable settings: `version`, `knowledge.path`, `repositories: [{name, ...}]`, `agent` (provider, optional command), `site` (port), `generation` (parallel hint).
  - `.akashic/config.local.yaml` — **gitignored.** Per-machine `name → absolute path` mapping.
  - `list`/`generate` merge the two; a name present in `config.yaml` with no local path is flagged (by `list` and `doctor`).
- **`.gitignore`** (written by `init`) ignores `.akashic/cache/`, `.akashic/logs/`, and `.akashic/config.local.yaml`.

### `init`
- Scaffolds the full tree (`services/ flows/ system/ adr/ entities/ glossary/`, `README.md`, `.akashic/`), writes `config.yaml`, runs `git init`, writes `.gitignore`, and makes an initial commit. Idempotent and safe in an existing repo.

### `attach` / `detach` / `list`
- `attach`: validate target is a Git repo → name = basename (or `--name`) → dedupe by resolved absolute path (re-attach updates in place) → enforce unique names → write name to `config.yaml`, path to `config.local.yaml`. Generates no documentation.
- `detach <name>`: remove from both config files; leave generated docs untouched; print cleanup hint. Deletes no repositories.
- `list`: print attached repos with an indicator for "no local path on this machine."

### Agent provider abstraction
- Interface: `generate(context)`, `is_available()`, `version()`.
- **Both providers implemented in v1.** The interface absorbs the real divergence:
  - **Claude Code:** interactive seeded session — `claude "<prompt>" --add-dir <repo> …` with cwd = knowledge repo.
  - **Codex:** interactive seeded session — `codex "<prompt>"` with its sandbox/writable-roots model exposing source repos read-side and the knowledge repo write-side.
- `is_available()` via `shutil.which` (respecting a config `agent.command` override); `version()` runs `<bin> --version`.
- **Invocation mode: interactive only in v1.** Akashic hands the TTY to the child process (inherit stdin/stdout) and waits for the session to exit. Permissions are approved by the user live — **no `--allowedTools` tuning, no `--dangerously-skip-permissions`, no output-stream parsing** in v1.
- **Success signal:** the session exits cleanly, then Akashic inspects `git status`/diff of the knowledge repo. Success = clean exit AND ≥1 changed file under the knowledge tree; warn on clean exit with no changes.

### Prompt library & composition
- `akashic/prompts/` ships **modular templates** (`generate_service.md`, `generate_flow.md`, `generate_system.md`, `generate_entity.md`, `generate_adr.md`) plus a master/wrapper template.
- At generate-time the prompt builder composes: master wrapper + relevant type templates + attached-repo list & paths + the knowledge-layout spec + HUMAN-preservation rules → **one master prompt string** inlined into the seeded session.
- Any template is overridable by a same-named file in `.akashic/custom-prompts/`.
- v1 generates **all** doc types uniformly (including `adr/` and `glossary/`). The "inferred vs. code-grounded" distinction (frontmatter marking + review workflow) is deferred.

### Document format
- Every generated file begins with frontmatter: `generated: true`, `type: service|flow|system|entity|adr|glossary`, `name:`, `sources: [<abs paths>]`, `updated_at: <ISO8601>`, `generator: akashic@<version>`.
- Below the generated body, files carry `<!-- HUMAN START -->` / `<!-- HUMAN END -->` markers. The agent is instructed to preserve their contents verbatim across regeneration (the agent has read access and preserves them itself — no Akashic-side merge engine in v1).

### Site layer
- `serve`: MkDocs (Material) dev server — navigation, search (lunr), Markdown rendering, Mermaid, responsive; port from `site.port`.
- `build-site`: `mkdocs build` → `dist/` (GitHub-Pages-ready).
- Editing in v1 = the user's own editor (local + Git). In-browser WYSIWYG/commit is deferred.

### `doctor` / `status`
- `doctor`: validate config schema, resolve each repo's local path (exists + is a Git repo), check `git` present, check the configured agent is installed and report its version, verify the knowledge repo is writable.
- `status`: after each `generate`, Akashic writes `.akashic/cache/state.json` (timestamp, provider, repos, duration). `status` reads that for "last generation time," runs `git status` for pending changes, and counts files per knowledge section.

## Testing Decisions

Good tests here assert **external behavior at the highest seam**, never implementation details. Because the risky, slow, non-deterministic part is the real agent CLI, the primary seam is the **`AgentProvider` interface**: a fake provider records the composed prompt/context it was handed and simulates writing files into the knowledge repo. This lets us test the entire `generate` pipeline deterministically without launching Claude/Codex.

Seams and what they let us test (highest → lowest), to confirm with the developer:

1. **CLI seam (highest).** Drive commands via Typer's `CliRunner` against temporary knowledge repos, with a **fake `AgentProvider` injected**. Covers `init` (scaffold + git init + commit, idempotency), `attach`/`detach`/`list` (naming, path-dedupe, unique-name error, split-config writes, missing-path flag), `generate` (prompt composition handed to provider, changed-file summary, warn-and-skip missing paths, abort-if-none, no-commit invariant), `doctor`, and `status`. This is the preferred seam — behavior a user actually observes.
2. **AgentProvider seam.** Contract tests for the interface: `is_available()`/`version()` against a stubbed binary on `PATH`; command/args composition for each provider verified without executing the real agent (assert the constructed invocation, not internals).
3. **Prompt builder (pure function seam).** Given a config + template set (including a `.akashic/custom-prompts/` override) → assert the composed master prompt contains the repo list/paths, layout spec, HUMAN-preservation rules, and that overrides win. Pure input→output, no I/O.
4. **Config load/merge (pure seam).** `config.yaml` + `config.local.yaml` → merged model; Pydantic validation errors are surfaced clearly; missing local paths represented explicitly.
5. **Workspace discovery (seam over the filesystem).** Walk-up from nested temp dirs finds `.akashic/`; `--knowledge`/`AKASHIC_HOME` overrides win; clear error when no knowledge repo is found.
6. **Git ops wrapper.** Thin subprocess wrapper tested against a temp repo (init creates a repo + initial commit; diff/status reflect changes). Assert observable git state, not the wrapper's internals.
7. **Site layer (`SiteProvider` seam).** `build-site` produces `dist/` from a fixture knowledge repo; `serve` binds the configured port. Keep MkDocs itself out of unit scope (treat as trusted dependency); test the adapter contract.

Prior art: none yet (greenfield). Establish the fake-provider + `CliRunner`-on-temp-repo pattern as the reference for future tests; prefer pytest with `tmp_path` fixtures.

## Out of Scope

- **`akashic sync`** — dropped from v1 entirely (pull/commit/push automation).
- **Headless / unattended generation** — the headless runner, scoped `--allowedTools`, `--dangerously-skip-permissions`, and JSON-event-stream parsing. (Consequently: scheduled cron/launchd/Task Scheduler runs and webhook/event-based triggers.)
- **Deterministic per-repo / per-doc-type orchestration** — v1 delegates all fan-out to the agent.
- **Deterministic merge engine** — human-section preservation is the agent's responsibility in v1.
- **In-browser editing / WYSIWYG / commit-from-browser / Git history UI.**
- **The FastAPI server and React frontend** (the `server/`/`frontend/` seams exist but only the MkDocs adapter ships).
- **"Inferred vs. code-grounded" doc marking / review workflow** for ADRs & glossary.
- Cloud backend, authentication, multi-user collaboration, knowledge graph, vector DB, semantic search, MCP server, incremental dependency graph, automatic architecture diagrams, IDE extensions.

## Further Notes

- **Known limitation — source-repo write protection:** exposing source repos to the agent grants it write access to those dirs, and file-writing tools are enabled so the agent can author docs. "Never modify source repositories" therefore rests on prompt instructions (write only under the knowledge repo) plus the user's live permission approval in the interactive session — it is not hard-enforced in v1. Recommend users generate against clean/committed source trees. Hard per-directory write scoping is a candidate for the future headless runner.
- **Known risk — hallucinated ADRs/glossary:** v1 generates these uniformly even though they are poorly grounded in code. Accepted for v1; mitigated later by the inferred-vs-grounded marking + review workflow.
- **`generation.parallel`** is effectively a hint in v1 since a single interactive session self-parallelizes; it gains meaning if/when deterministic orchestration lands.
- **Success criteria (v1 done when):** a developer can (1) init a knowledge repo, (2) attach source repos without moving them, (3) generate structured docs via their chosen agent, (4) have manual HUMAN-section edits preserved across regeneration, (5) browse and edit docs locally, (6) keep the whole knowledge base in Git with full history.
